# ТЗ: «Оборудыш» — Telegram Mini App

Версия: 1.0 (draft) · Дата: 02.07.2026
Заменяет черновик ТЗ; сведено с фактическим функционалом бота @mediaoborudka_bot (по логам обновлений и осмотру).

---

## 0. Суть проекта

Перевод бота «Оборудыш» (бронирование съёмочного оборудования и аудитории 626 Media BMSTU) с кнопочного интерфейса на **Telegram Mini App** (web-app внутри Telegram).

**Модель — полный переезд:**
- Всё взаимодействие пользователя и админа происходит в Mini App.
- Бот остаётся как канал доставки: уведомления, напоминания, карточки заявок в админ-чате (с кнопкой «Открыть в приложении»), команда `/start` открывает Mini App.

**Почему Mini App, а не кнопки:** мастер заявки с корзиной и календарём занятости невозможно удобно сделать на inline-кнопках; сейчас заявка собирается цепочкой из 10+ сообщений, редактирование — заново. В web-app это один экран с нормальной навигацией «назад/далее».

## 1. Пользователи и роли

### 1.1. Кто пользуется
- Активисты, стажёры, главы, менторы Media BMSTU
- Главы медиа-направлений организаций СО/ССФ (бронь от СО/ССФ — **только от главы** медиа-организации)

### 1.2. Профиль пользователя (User)
| Поле | Примечание |
|---|---|
| id | Telegram ID (получается из initData автоматически) |
| ФИО | вводится при регистрации |
| username | из Telegram |
| Организации | Media BMSTU и/или **СО/ССФ** (СО = ССФ = ССО = ССК — одна роль). При выборе СО/ССФ конкретная организация выбирается из списка (Студсовет, ССФ, ССО, ССК…) + «Другое» с ручным вводом — как в текущем боте |
| Отделы | только для MB: Фото, Видео, Дизайн, СММ, Прод, VFX. У СО/ССФ отделов нет |
| Статус (роль) | Три роли, по возрастанию доступа: **СО/ССФ → стажёр → активист**. Назначает админ при верификации; роль **никогда не понижается автоматически** (если есть выше прошлой — остаётся). Ролей «глава»/«ментор» в системе нет: доступ уровня главы — скрытый пароль (§3.4) |
| Статус верификации | не верифицирован / ожидает / верифицирован / отклонён (таймер) / заблокирован |
| Порядочность | **v2** — числовой рейтинг; менять могут только глава Media Tech и заместители |

### 1.3. Уровни доступа в системе
| Уровень | Права |
|---|---|
| Пользователь | брони оборудования и 626, свои заявки, переписка с куратором |
| Админ | всё выше + заявки без куратора, свои курируемые заявки, работа с заявкой (кураторство → согласование → выдача → возврат) |
| Старший админ | всё выше + все заявки, пользователи (просмотр, верификация, блокировки, чёрный список), согласование броней 626, добавление/удаление админов, добавление оборудования в рентал, массовые рассылки, статистика, рассмотрение спорных возвратов |

Админ может сам бронировать оборудование/626 и **может взять свою заявку в обработку**.

### 1.4. Регистрация и верификация
1. Первый вход в Mini App → экран регламента. Без нажатия «Согласен с регламентом» дальше пройти нельзя (показывается один раз).
2. Заполнение профиля: ФИО, организации, отделы, статус.
3. **Верификация — последний этап регистрации:**
   - Организация Media BMSTU → автоматическая сверка с общей таблицей участников → верифицирован сразу.
   - Есть СО/ССФ → заявка на верификацию уходит старшим админам.
4. Экран у старшего админа — назначение роли (активист/стажёр) и три действия:
   - **Верифицировать** → доступ открыт с назначенной ролью;
   - **Отклонить** → таймер 3 часа до следующей попытки;
   - **Заблокировать** → модалка с причиной и сроком блокировки.
5. До верификации приложение показывает только экран «ожидает проверки». Заблокированный видит: «Вы в чёрном списке оборудыша. Для уточнения деталей — @Kyuller».

## 2. Оборудование (Equipment)

| Поле | Примечание |
|---|---|
| Категория / подкатегория | сейчас 11 категорий (Камеры, Объективы, Свет, Звук, …) |
| Короткое название | показывается в списках |
| Полное название | раскрывается по тапу на позицию |
| Количество экземпляров | у позиции может быть несколько единиц |
| Инвентарный № МГТУ | требует уточнения наличия на проде |
| Теоретический № экземпляра | у позиций с несколькими экземплярами каждый имеет номер 1..N (в Google-таблице — отдельная строка на экземпляр). Пользователь номер **не видит**; при выдаче система предлагает админу **минимальный свободный** (свободны 3, 5, 6 → предлагает 3), админ может выбрать другой свободный |
| Внутренний ID | для системы распознавания (QR в v2) |
| У кого сейчас | вычисляется из активных заявок (журнал выдачи) |
| Уровень доступа (Допуск из таблицы) | «Стажер MB + СО/ССФ» (все) / «Активист Media BMSTU» / «НЕ ВЫДАЕТСЯ» / уровень главы — в UI подписывается «🔒 по паролю», слово «глава» пользователям не показывается |
| Фото | локальный файл `prototype/img/<Короткое имя>.jpg` (имя = короткое имя позиции, точно как в каталоге); нет файла — SVG-иконка категории |
| Блокировка категории | админ может заблокировать категорию на срок (например «Хранение данных» на полгода) — позиции не выбираются |

Источник данных — существующая таблица (сейчас генерится equipment_data.py из CSV/Google Sheets). В v1 админ добавляет позиции через экран «Добавить оборудование» (только старшие).

**Занятость** считается по пересечению интервалов дат активных заявок: `свободно = всего − забронировано на пересекающиеся даты`. Занятые на выбранные даты позиции выбрать нельзя.

## 3. Заявка на оборудование

### 3.1. Поля
- Автор (профиль), даты «взять — вернуть», список позиций с количеством, название мероприятия, комментарий
- Куратор (id админа), комментарий выдачи, комментарий возврата
- Чекбокс «ответственный за медиа» — открывает выбор оборудования уровня «активист» любого направления
- Служебное: статус, таймстемпы переходов, кто совершил действие

### 3.2. Жизненный цикл (единый workflow)
Сведён из текущих статусов бота и куратор-модели черновика:

```
 Новая ──► Назначен куратор ──► Согласована ──► Выдана ──► Возврат на проверке ──► Закрыта (архив)
   │              │                   │             │               │
   │              ▼                   ▼             │               ├─► Возврат отклонён ──► (снова Выдана)
   │         Отклонена           Отклонена          │               └─► На проверке у старших ──► Закрыта
   │                                                └─ напоминания о сдаче
   ├─► Отменена пользователем (до согласования можно менять и удалять)
   ├─► ⏳ Не рассмотрено в срок (авто, 3 дня без решения)
   └─► ⏳ Срок получения истёк (авто: не выдали до даты возврата)
```

Правила:
- При создании заявки позиции **резервируются в календаре** (если свободны на эти даты).
- До согласования пользователь может редактировать (каждый пункт отдельно, добавлять/убирать позиции) и удалять заявку; заявок может быть несколько одновременно.
- «Стать куратором» — первая кнопка в карточке для админа. После нажатия открывается переписка с заявителем, статус → «Назначен куратор».
- Согласовано/Отклонено → далее Выдано/Отклонено. При выдаче можно добавить комментарий.
- Возврат: если комментарий возврата пуст — заявка закрывается в архив; если есть комментарий — статус «На проверке у старших», после их решения — в архив.
- QR-сканирование при выдаче/возврате — **v2**; в v1 админ отмечает позиции чек-листом вручную (это же fallback «если QR отпал»).
- Любая автоотмена: карточка в админ-канале обновляется, кнопки убираются, оборудование освобождается в календаре, пользователю уходит сообщение.

### 3.3. Ограничения и подсказки
- Заявка минимум **за 2 дня** до мероприятия (поздние — на усмотрение команды).
- Карты памяти / SSD / носители — **отдельной заявкой**.
- Запрет времени сдачи на вечер субботы и воскресенье (чтобы возврат не прилетал в выходной вечер) — валидация в календаре.
- Слоты времени: 09:00–21:30 с шагом 30 мин; время выбирается из полного списка слотов (не из «быстрых» пресетов).
- Бронь может пересекать границу месяца (взял в июле — вернул в августе); календарь листается по месяцам, занятость проверяется на весь период.
- Оборудование уровня «активист» доступно только активистам (или по чекбоксу «ответственный за медиа»).
- **Проверка комплектности** (как `camera_kit` в текущем боте): при камере без объектива, объективе без камеры, свете без аккумов — предупреждение в корзине и на шаге деталей (не блокирует отправку).

### 3.4. Скрытый доступ по паролю
- Позиции уровня главы и **заблокированные категории** можно взять, введя пароль **«БИПКИ»** (регистр не важен). Пароль знают главы отделов; в UI подпись — «🔒 по паролю», без упоминания глав.
- Верный пароль → позиция добавляется в корзину / категория открывается до конца заявки.
- В карточке заявки у админа — пометка **«🔒 Взято по паролю»**.

## 4. Аудитория 626

### 4.1. Заявка на 626
- Профиль, цель съёмки/записи, дата через календарь (с перелистыванием месяцев), время — слоты с подсветкой 🟢 свободно / 🔴 занято, пересечения с активными бронями запрещены
- Чекбоксы: **хромакей**, **циклорама**, **чёрная ткань**, «нужна помощь Media Tech» + пункт **«Другое»** со свободным текстом (попросить что-то своё)
- При бронировании 626 доступен список свободного оборудования уровня «активист»
- **Кураторство**: после согласования старшими любой админ может нажать «Стать куратором» — открывается переписка заявителя с куратором (как в заявках на оборудование)

### 4.2. Жизненный цикл
```
Новая ──► Согласована (старшим) ──► (куратор из админов, съёмка) ──► Сдача на проверке ──► Закрыта
  │              │                                                        │
  ├─► Отменена   ▼                                                        └─ комментарий ≠ пуст → на проверку старшим
  │          Отклонена
```
- Согласовывать брони 626 могут **только старшие админы**; кураторство после согласования берёт любой админ.
- Сдача: пользователь прикрепляет **3 фото** аудитории.
- После закрытия пользователю: «Спасибо, приходите ещё».
- Напоминание о сдаче после окончания брони.

## 5. Экраны Mini App

### 5.1. Пользователь
1. **Онбординг — первый экран приложения**: регламент → «Согласен» → профиль (ФИО, организации, отделы) → автоверификация по Google-таблице участников MB; для СО/ССФ — «ожидает верификации». Отдельной кнопки «регистрация» на главном нет — только пункт «Регламент» для чтения.
2. **Главный экран**:
   - шапка: аватар из Telegram, имя с галочкой ✓ (верифицирован — без текста), @ник, кнопка ⚙️ «изменить данные аккаунта»
   - роли — чипсами: организации, отделы, статус (активист/стажёр/глава/ментор); если человек админ/старший — тоже чипс
   - без даты/времени и декоративных элементов
   - крупная главная кнопка «📦 Забронировать оборудование», ниже парой поменьше — «🏛 Студия 626» и «📘 Регламент»
   - «Мои активные брони» — только свои; заголовок карточки — состав («R8 +2»), подзаголовок — мероприятие и даты; тап → детали
   - «История броней» — с фактическим временем: «взял 25.06, 09:04 · вернул 26.06, 20:41»
   - «Задать вопрос» — свободный текст, можно анонимно, **без лимита** на количество обращений
3. **Мастер заявки** (шаги с «назад/далее», прогресс сверху; «Назад» в шапке ведёт по шагам, а не выкидывает из мастера):
   1. даты и время — календарь с подсветкой загруженности (🟢 0–1 / 🟡 2–4 / 🟠 5–6 / 🔴 7+ заявок), **перелистывание месяцев**, диапазон может пересекать месяц; время «взять/вернуть» — из полного списка слотов
   2. каталог — **чипсы категорий** для быстрого переключения (позиций много), поиск по позициям **и по названиям категорий** (выбранный чипс сужает поиск), у позиции: **фото** (локальный файл `prototype/img/<Короткое имя>.jpg`, фолбэк — SVG-иконка), короткое имя, «свободно N из M», уровень доступа, счётчик количества; тап → полное название и фото. Иконки категорий — единый SVG-набор (line-art), не эмодзи
   3. **корзина** — всегда видна (счётчик), можно убрать/поменять количество; здесь же предупреждения о комплектности
   4. детали: **вся корзина** с кнопкой «Изменить набор» (возврат к каталогу), мероприятие (обязательно), комментарий (**необязательно**), чекбокс «ответственный за медиа»
   5. итоговое подтверждение → «Отправить» → возврат на главный экран
4. **Мои заявки**: карточка со статус-таймлайном, редактирование/отмена (до согласования), переписка с куратором; когда пользователь пишет — **куратору уходит уведомление от бота**.
5. **626**: мои брони (карточка с таймлайном, куратором и перепиской), календарь + слоты, чекбоксы + «Другое», отмена, сдача (3 фото); после отправки — возврат на главный экран.

### 5.2. Админ (роли разделены: у старшего — своя панель без заявок)
**Панель админа:**
1. Очереди: заявки без куратора / мои курируемые / на сегодня.
2. **Карточка заявки**: «Стать куратором» → «Согласовано/Отклонено» → «Выдано (комментарий)» → чек-лист выдачи позиций (ручной; QR — v2) → «Возврат (комментарий)» → закрытие или к старшим. Переписка с заявителем прямо в карточке; когда заявитель пишет — админу приходит уведомление от бота.
3. 626: согласованные старшими брони без куратора — кнопка «Стать куратором» + переписка.

**Панель старшего админа** (заявки на оборудование сюда не входят — их ведут админы):
1. Согласование 626: Согласовать/Отклонить; после согласования бронь появляется у админов для кураторства.
2. **Пользователи**: поиск по ФИО/организации, профиль (в v2 — правка порядочности), очередь верификации (Верифицировать/Отклонить/Заблокировать), чёрный список.
3. **Оборудование**: добавить позицию в рентал (с фото), блокировка категорий на срок.
4. **Рассылка**: всем / выбранным. **v1.1**
5. **Статистика**: см. §7. **v2**

### 5.3. Принципы интерфейса
- Telegram-нативно: цвета из `themeParams` (`--tg-theme-*`), MainButton внизу для главного действия шага, BackButton Telegram для «назад» (внутри мастера — по шагам).
- **Приоритетная тема — тёмная** (графит + оранжевый акцент Media BMSTU); светлая поддерживается через themeParams.
- Аватар и ник берутся из Telegram (photo_url из initData / getUserProfilePhotos у бота).
- Из любого экрана можно вернуться на главный; после отправки заявки/брони пользователь возвращается на главный экран.
- Мобильная вёрстка (виджет ~380px), тап-таргеты ≥ 44px.
- Никаких логинов/паролей — авторизация прозрачна через `initData`.

## 7. Статистика (v2)

1. Самое востребованное оборудование
2. Самый загруженный месяц по броням оборудования
3. Самый загруженный месяц по броням 626
4. Отдел-лидер по просрочкам сдачи
5. Активность админов по заявкам оборудования (С/В/О/П)
6. Активность админов по 626 (А)
7. Список людей с самой низкой порядочностью

## 9. Техническая часть

### 9.1. Как устроен Telegram Mini App
- Обычное веб-приложение по HTTPS, открывается внутри Telegram (кнопка меню бота / inline-кнопка / `/start`).
- **Авторизация**: Telegram передаёт `initData` (id, username, подпись). Бэкенд проверяет подпись HMAC-ом от токена бота — регистрация/логин не нужны.
- Тема: `Telegram.WebApp.themeParams` → CSS-переменные, автоматически светлая/тёмная.
- UI-API: MainButton, BackButton, HapticFeedback, `showScanQrPopup` (v2), попапы подтверждений.
- Пуши из Mini App невозможны → все уведомления шлёт бот сообщениями (§6).

### 9.2. Рекомендуемый стек (для этапа разработки)
- **Бэкенд**: Python — FastAPI (REST API для Mini App) + aiogram 3 (бот: уведомления, карточки в админ-чат). Один процесс/репозиторий.
- **БД**: SQLite на старте (как сейчас), миграция на Postgres при необходимости; SQLAlchemy + Alembic, чтобы переход был безболезненным.
- **Фронтенд**: выбор на этапе разработки; прототип стек-агностичен (vanilla HTML/CSS/JS). Кандидат по умолчанию — Vite + React + `@telegram-apps/sdk`, но можно остаться на vanilla, если объём v1 урежется.
- **Хостинг**: любой VPS + Caddy/nginx (HTTPS обязателен); на время разработки — туннель (cloudflared / ngrok).
- **Планировщик**: APScheduler для напоминаний и автоотмен.

### 9.3. Миграция
- Данные текущего бота (sqlite: users, checkouts) переносятся скриптом; профили со старым «СО/ССФ без организации» помечаются как требующие уточнения (как уже сделано в боте).
- Каталог оборудования — импорт из существующей таблицы.

## ⚠️ 10. Безопасность — сделать сразу

В разосланных `main.py` (строки 35, 40) и `bot_fixed.py` (строка 30) **захардкожены токены бота**, включая боевой. Файлы пересылались в Telegram → токены считать скомпрометированными: отозвать через @BotFather (`/revoke`), новые хранить только в переменных окружения / `.env` вне репозитория. Для разработки и тестов Mini App использовать **отдельного тестового бота с новым токеном** — боевой токен в тестах не использовать никогда. Список Telegram ID админов — в новой системе держать в БД (в `bot_fixed.py` уже есть `SENIOR_ADMIN_IDS` — перенести).
