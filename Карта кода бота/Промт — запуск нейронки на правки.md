# 🤖 Промт — запуск нейронки на правки и фичи

Один большой промт, который даёшь ИИ-ассистенту (Claude Code, Cursor, и т.п.), открытому в папке проекта `D:\Media BMSTU\Оборудыш`. Он сначала изучает контекст, потом делает правки и проверяет.

> [!info] Как пользоваться
> 1. Открой ассистента в корне проекта (чтобы у него был доступ к файлам).
> 2. Скопируй весь блок ниже.
> 3. При желании отредактируй раздел «ЗАДАЧИ» — оставь только то, что нужно сейчас.

---

## 📋 Мастер-промт (копировать целиком)

```
Ты — инженер, продолжаешь проект «Оборудыш» (Telegram Mini App для бронирования съёмочного
оборудования и студии 626 медиацентра 626 Media BMSTU, МГТУ им. Баумана).

ШАГ 0 — ИЗУЧИ КОНТЕКСТ ПЕРЕД ЛЮБЫМИ ПРАВКАМИ (прочитай целиком, не по диагонали):
- CLAUDE.md — главный источник контекста (роли, статусы, договорённости, что уже сделано).
- HANDOFF.md — технические детали и хронология итераций.
- Папку «Карта кода бота/» — разбор кода по слоям (начни с «00 Карта бота.md»;
  особенно: «Слой БД», «API-эндпоинты», «Планировщик», «boot_payload — сборка ответа»,
  «Данные — каталог и списки», «Гайд — 4 правки с промтами»).
- bot/main.py — весь бэкенд (aiohttp + aiogram 3 + SQLite, ~1730 строк).
- prototype/index.html — весь фронт (vanilla JS/CSS). prototype/catalog.js — каталог.
Сначала кратко перескажи мне своими словами архитектуру и то, как проходит одно действие
(фронт → API → БД → уведомление), чтобы я убедился, что ты понял. Потом приступай.

ЖЁСТКИЕ ПРАВИЛА (нарушать нельзя):
- Python 3.8 на проде: без синтаксиса 3.9+ (str.removeprefix, Path.is_relative_to, X|Y в аннотациях, и т.п.).
- Каждый мутирующий @auth-эндпоинт заканчивается: return web.json_response(boot_payload(uid)).
- Новую таблицу/колонку заводишь в init_db() И дублируешь в _migrate() (иначе прод-база упадёт).
- Новый роут регистрируешь в main() через app.router.add_post(...).
- Уведомления: notify(uid,text)/notify_seniors(text). Время now_str() -> "%d.%m, %H:%M", таймзона MSK.
- Права: is_admin(uid)/is_senior(uid); старший ⊇ админ.
- Секреты/токены не трогать, не логировать, не хардкодить. DEV_USER_ID на проде не включать.
- Строки в onclick на фронте прогонять через escJs().
- Держись стиля соседнего кода; не рефактори лишнее; не меняй дизайн без явной просьбы.

ЗАДАЧИ (полные спецификации и промты по каждой — в «Карта кода бота/Гайд — 4 правки с промтами.md»,
сверься с ней):
1) Админ дня в ежедневной сводке (daily_digest): кто из админов сделал больше всех действий за день.
   Нужно завести журнал действий (таблица actions), логировать админские действия в api_req_action/
   api_626_action, в сводке выводить лидера за сутки. Учти: текущая история (_push_hist) для этого
   непригодна (нет id админа и года).
2) В сводке перечислять заявки на завтра с номером и куратором: выдачи (approved, dfrom_iso=завтра)
   и возвраты (issued, dto_iso=завтра) — списком «ID … — куратор …».
3) Кнопка (в seniorHub) «Обновить БД / пересверить людей»: новый эндпоинт /api/resync (только старший),
   пересверяет всех agreed=1 по mb_ok/org_ok; пропал из списка → verified='pending' + уведомить старших;
   если роль в источнике изменилась → обновить (зависит от задачи 4).
4) Автозаполнение отделов и роли по Telegram-нику: нужен богатый источник (username;name;departments;role) —
   доработать make_members.py / новый парсер + loader directory() в main.py + подстановка в api_register.
   Задачу 3 (авто-роль) делать после 4.

ПОРЯДОК: сначала независимые 1 и 2, затем 4, затем 3. Для каждой задачи:
- сначала покажи план (какие функции/файлы тронешь) и жди моего «ок», если задача крупная;
- вноси минимальные точечные диффы;
- после правки — самопроверка (см. ниже).

ПРОВЕРКА (обязательна для бэкенда):
- Локальный DEV-режим (бот не стартует, любой запрос = один uid):
  запуск полным путём к Python из CLAUDE.md, переменные:
  DEV_USER_ID=111 ADMIN_IDS=111 SENIOR_ADMIN_IDS=111 PORT=8739 ADMIN_CHAT_ID=0.
  Бей по http://localhost:8739/api/... . Для ролевых веток вставляй фейковых юзеров/заявки прямо в sqlite.
  Планировщик прогоняй вручную через /api/dev/tick.
- Фронт проверяй в preview (демо на моках); SRV-режим симулируй applyBoot({...}).
- После тестов останови сервер и удали bot/oborudka.db и bot/backup/.

ФОРМАТ ОТВЕТА:
- Короткое резюме архитектуры (шаг 0).
- По каждой задаче: план → диффы → что и как проверил (с результатом).
- В конце: что заменить/дописать при деплое (файлы, новые переменные .env, нужен ли перезапуск),
  и что осталось на ручную проверку в реальном Telegram.
```

## 🔧 Мини-версия (если задача одна)

Если нужна только одна правка — не грузи всё, дай короткий промт:

```
Проект «Оборудыш», ты продолжаешь его. Сначала прочитай CLAUDE.md и в «Карта кода бота/» заметки
«00 Карта бота», «<нужный слой>» и «Гайд — 4 правки с промтами». Соблюдай правила оттуда
(Python 3.8; @auth-эндпоинт возвращает boot_payload; новая колонка/таблица — в init_db И _migrate;
роут — в main(); секреты не трогать). Затем выполни задачу N из «Гайд — 4 правки с промтами»:
<вставь её промт>. Проверь в DEV-режиме и отчитайся диффами + результатами проверки.
```
